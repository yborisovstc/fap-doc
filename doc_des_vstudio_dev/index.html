Introduction

Discrete Events Systems (DES) is the effective way for systems modelling and simulation. There was DES framework introduced, based on specific language for systems structure and states transitions (ref [FAP_DOC_DSA_IN_SWD], [FAP_DES_EMB_TRANS]).

The common approach for such a framework based development is creating the set of tool that helps specify systems, visualizing, tracing, debugging etc. The first step here can be creating of lightweighted integrated development environment (IDE) containing some initial set of tools. This article outlines the attemplt of creating such a IDE application.

Another important feature is visualization of system run. There will be one of possible solutions considered in the article, that provides simple native interface for DES system for access to environment widget system.

Tools and features

Let's consider the initial set of features and tools supported by DES IDE. The list can be as:

  • Creating DES enviroment
  • Creating the system from the spec file
  • Visualization - view system spec, editing
  • Displaying the system log
  • Running in continuous and stepping modes

This set of features allows user to perform complete development of simple system. The typical development cycle would be as:

  • IDE creates DES environment by default with empty root system.
  • User uses DES visual representation to create his system within root.
  • User test the system by running or stepping it.
  • User checks the system evaluation and errors from the log displayed.

DES visual representation language (DESVL)

For implementing the system visualization functionality of DES it is important to have simple and convenient visual language for such a representation.

There are some variants of DES visual representation. For instance the kind of tree-based form would be sutable because the native structure of DESL is tree based. We will focus on visual form that shows all elements directly, including connections. This means that it will be representation for single level of system's hierarchy.

DES generic language (DESL) describes the system via specifying of system's structure and transitions, thus DESVL is just another form of system representation. So DESVL has to generally correspond generic language in system description elements.

The following notations are proposed for base element of visual language (also DESL elements are given for comparition purpose).

State

State element consists of header and body, where header contains name and type, but body contains inputs, transition, and initial value.

Figure 1. 


Subsystem or component

Component is a subsystem within the given system. Component detailed structure is not represented in this element in order to make the system's visual description is more compact. It is IDE responsibilty to secure displaying the component in detail as a system in case of the user request.

So component contains only parts that are important to show the system structure - name, type, inputs, and outputs.

Figure 2. 


System

System's element of DESVL represents the whole system exlcuding deeper layers of hierarchy. The element contains states, components, transtion, inputs, outputs, and connections.

Mapping of visual representation to the run-time model

We can consider two approaches for relation of visial representation to system model.

  • Visual representation corresponds to system's DESL specification.
  • Visual model represents run-time system's.

DESVL to specification

In this case the direct mapping between visual and general specification are used. Each DESVL element on the visual specification results in generation corresponding DESL specification node. So the chain of creating DES runtime structure is as: DESVL-> DESL->Runtime.

The weakness of this approach is that visual representation shows the system in initial state only but not in dynamic. It is not possible to see of how the system structure changes when running.

DESVL to DES run-time

With this approach the visual representation corresponds to run-time DES. Each element of visual representation corresponds to element of run-time structure. So it makes possible to show the system in all the steps of evaluation. Also the state vector can be shown. The general DESL specification can be created from run-time in turn. So the relation schema is as: DESVL -> Runtime -> DESL

We will focus on the second approach that seems more preferrable basing on this analysis.

IDE design outline

Simple IDE architecture corresponds to standard Model-View-Controller (MVC) design template. The diagram below outlines the structure of application variant based on GTK framework.

Figure 4. 


The main elements of architecture are as:

  • Application class plays role of Controller. It is responsible for creating the model, view, and main window.
  • CAE_Env class (core environment of FAP system) acts as Model. The class creates the system hierarchy from the specification given by Controller. The environment also give an access to root system of hierarchy.
  • CagProxy class is View proxy for the system. The proxy implements the proxy interface MAE_Opv, thru which interacts to the system.

The initialization interactions are as:

  • User opens application, Controller is created.
  • Controller creates Model and pass to it default specification of the system.
  • Model creates the system from default specification and makes the root system available.
  • Controller create View and "bind" it to the root system.
  • User is able to navigate thru the whole hierarchy of the system - View interacts to the binded subsystem in order to move the cursor within the hierarchy tree.

For source codes of the projects of FAP IDE, and View proxy refer the repositories [FAP_STUDIO_GIT] and [FAP_BVP_GIT]

Example: "Hello world"

Let us test IDE with simple DES application, a kind of C Hello world. This application will just implements simple incrementor.

  • The system contains only one state of type TInt named "Incrementor".
  • The state input is state's own output, thus the state updates constantly.
  • Transition function is just incrementing of state.

To create the application:

  • Open specification template "Empty" - IDE will create DES environment with empty root system.
  • Add the state - just drag and drop the icon "state" from toolbar to root system area - IDE will create state with random name and of default type "StInt" - integer.
  • Rename the state - edit state name in place, press "Enter" to complete editing.
  • Add input to the state - drag and drop button with input icon from toolbar to the state - IDE will create input with random name. Rename input to "inp1" by editing in place.
  • Add transition "add inp inp1 1" for incrementing state - enter text into transition area.
  • Connect input "inp1" to state's output - drag input label to output area.

Visualization module

There are some possible approaches for DES visualization. One of them is creating specific subset of DESTL for access to environment's widget system. In this case the event system would include the proper DESTL commands to transitions in order to display the status. Thou being seemed direct way to widget subsystem this approach cause to overhead of transition with command that doesn't relate to transition really.

Therefore we consider another approach where dedicated events system is used for interacting to widget system. So this dedicated visualization DES acts as a proxy between DES and real widgets.

Figure 10. 


The visualization system itself need to access to widgets via native interface of widgets system. The system transitions have to be able to use this interface so the transitions should be written on C, C++. This can be done by creating dedicated plugin for DES framework in order to allow DES environment load transitions functions and "attach" them to visualization DES states. Visualization DES in whole can be delivered as DES framework module so it can be loaded in any DES system and used properly.

Let us look at the simple example of such a visualization module.

Figure 11. 


In this example there are only three systems that "wrap" native widgets:

  • Window - wraps main window widget
  • Fixed - wraps container that allows external control of positioning the childs
  • Drawig_Area - wrapper for simple widget that allows direct drawing into it.

This simple visualization module can be used for instance for displaying status of DES simulating 2D physic models. We can use it to make the example "Rock" above vusualized.

Discussion

We have presented simple IDE for DES basing on visual representation languge (DESVL).

We outlined the approaches to visial representation binding to DES run-time model and consider direct representation as more suitable for IDE.

We described the architecture for simpe IDE application and main interactions between the general components.

Also the approaches for DES visualization were discussed and the structure of visualization "proxy" module was considered.

Finally we created couple of simple examples that show the typical procedure of developing and testing DES application in IDE.
